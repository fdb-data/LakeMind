# LakeMindMonitor 改造方案

> **定位**：LakeMind 面向人类的专业可观测性控制台。只读展示 + Steward 对话。
> 投资人演示第一屏。必须专业、信息密度高、实时反映系统状态。

---

## 1. 现状问题

| 问题 | 严重度 |
|------|--------|
| 5 个页面全部用 `<pre>` 输出原始 JSON，无法快速获取信息 | 🔴 |
| Dashboard 只显示 4 个 MCP/Steward 连通性，不显示 11 引擎健康、12 容器状态、资产计数 | 🔴 |
| Asset 页展示原始 JSON，无知识库/技能/记忆的结构化展示 | 🔴 |
| Data 页只有 `data_list_tables`，不展示向量表、S3 对象、KV、图 | 🔴 |
| Admin 页展示原始 JSON，无表格、无状态徽章 | 🔴 |
| Chat 页能发消息但 Steward 是关键词路由（非 LLM），无快捷指令、无历史 | 🟡 |
| Steward `inspect()` 有 bug：检查 `v != "ok"` 但 Server 返回 `true/false`，导致永远报错 | 🔴 |
| 无自动刷新、无加载态、无错误重试 | 🟡 |

---

## 2. 设计目标

1. **Dashboard**：一屏看清全局——12 容器状态 + 11 引擎健康 + 资产计数 + Ray 集群。
2. **Asset**：认知资产结构化展示——知识库（概念数/类型分布）、技能（名称/描述/标签）、记忆（总数/最近条目）、本体（节点/边）。
3. **Data**：数据引擎概览——Iceberg 表、LanceDB 向量表、S3 对象、Valkey KV、PG 图。
4. **Admin**：管理只读——租户/用户/Token/资产类型，表格 + 状态徽章，**无任何写操作按钮**。
5. **Chat**：真正连 Steward，专业对话 UI + 快捷指令 + 巡检面板。
6. **整体**：深色主题、自动刷新、加载骨架屏、错误降级、响应式布局。

---

## 3. 技术选型

| 层 | 选型 | 理由 |
|----|------|------|
| BFF | **Express**（保留现有 server.js，扩展端点） | 已工作，无需换语言 |
| 前端 | **Vue 3 + Vite + TypeScript + Element Plus + ECharts** | 专业组件库 + 图表，投资级 UI |
| 状态 | **Pinia** | Vue 3 标准状态管理 |
| HTTP | **axios** | 拦截器、超时、重试 |
| 容器 | **多阶段 Dockerfile**：node:20-alpine 构建前端 → node:20-alpine 运行 Express + 静态资源 | 单容器，无 Python 依赖 |

> 不引入：Nuxt（过重）、React（生态不匹配）、Tailwind（Element Plus 已够）。

```
│ │ [查看知识库] [查看技能]     │  │ │ 问题: 无        │
│ │ [查看记忆] [查看数据表]     │  │ │                 │
│ └─────────────────────────────┘  │ │ ┌─ 引擎详情 ──┐ │
│                                   │ │ │ ✅ 对象存储 │ │
│ You: 系统健康吗？                 │ │ │ ✅ 表格式   │ │
│                                   │ │ │ ✅ 向量     │ │
│ Steward: 11 引擎全部正常...        │ │ │ ...         │ │
│                                   │ │ └─────────────┘ │
│ [输入消息...]              [发送] │ │                 │
└───────────────────────────────────┘ └─────────────────┘
```

**功能**：
- 消息气泡：用户右蓝色，Steward 左绿色，Markdown 渲染
- 快捷指令：一键发送预设消息（健康/用户/租户/知识/技能/记忆/数据表）
- 连接状态：实时检测 Steward `/health`，绿点/红点
- 巡检面板：点击"运行巡检"调 Steward `/inspect`，展示引擎健康矩阵
- 消息历史：localStorage 持久化，页面刷新不丢失
- 打字动画：Steward 回复时显示"正在输入..."

**Steward 修复**（必须）：
- `inspect()` 的 `analyze` 节点：`v != "ok"` → `v is not True`（Server 返回 bool 非 str）
- `chat()` 的 data 意图：`data_list_tables` → `list_tables`（工具名不匹配）

## 5. BFF API 扩展

现有 11 个端点保留，新增以下：

| 方法 | 路径 | 说明 | 数据源 |
|------|------|------|--------|
| GET | `/api/dashboard/overview` | **聚合端点**：容器状态 + 引擎健康 + 资产计数 + 数据计数 + 平台信息 | 并行调 6+ MCP |
| GET | `/api/asset/knowledge/:kb` | 知识库详情（概念列表） | AssetMCP `lake://knowledge/{id}` |
| GET | `/api/asset/memory/list` | 记忆列表（分页） | AssetMCP `list_memory` tool |
| GET | `/api/data/vectors` | 向量表列表 | DataMCP `lake://vectors` resource |
| GET | `/api/data/s3` | S3 对象列表（?prefix=） | DataMCP `s3_list` tool |
| GET | `/api/data/kv` | KV 键扫描（?prefix=） | DataMCP `kv_scan` tool |
| GET | `/api/data/graph` | 图概览 | DataMCP `graph_query` tool |
| GET | `/api/data/tables/:ns/:t` | 表 schema | DataMCP `describe_table` tool |
| GET | `/api/admin/nodes` | 节点状态 | AdminMCP `get_node_status` tool |
| GET | `/api/admin/metrics` | 平台指标 | AdminMCP `get_metrics` tool |
| GET | `/api/admin/asset-types` | 资产类型 | AdminMCP `list_asset_types` tool |

**`/api/dashboard/overview` 聚合策略**：
```js
// 并行发起所有请求，单个失败不阻塞整体
const [containers, engines, knowledge, skills, memory, ontology,
       tables, vectors, tenants, users, tokens, nodes, metrics] =
  await Promise.allSettled([
    probeContainers(),      // 12 容器 /health
    adminHealth(),          // 11 引擎
    assetResource('lake://knowledge'),
    assetResource('lake://skills'),
    assetResource('lake://memory'),
    assetResource('lake://ontology'),
    dataCall('list_tables', {}),
    dataResource('lake://vectors'),
    adminCall('list_tenants', {}),
    adminCall('list_users', {}),
    adminCall('list_tokens', {}),
    adminCall('get_node_status', {}),
    adminCall('get_metrics', {}),
  ])
// 每个字段: { status: 'fulfilled'|'rejected', value: ... }
```

## 7. 容器化

```dockerfile
# 阶段 1：前端构建
FROM node:20-alpine AS frontend
WORKDIR /fe
COPY frontend/package.json frontend/package-lock.json ./
RUN npm ci
COPY frontend/ .
RUN npm run build          # → /fe/dist

# 阶段 2：运行时
FROM node:20-alpine
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm install --production
COPY server.js ./
COPY --from=frontend /fe/dist ./public/
EXPOSE 3000
CMD ["node", "server.js"]
```

构建时间预估：~30s（npm ci + vite build），远优于 server-api 的 10min。
