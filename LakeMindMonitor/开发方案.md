# LakeMindMonitor 开发方案（MVP）

> 依据《LakeMind整体技术方案.md》§2.2/§2.3/§7.2/§7.3 与《LakeMind简介.md》制定。
>
> **定位**：面向人类的只读可观测性控制台 + Steward 对话窗。极轻——仅装 MCP 客户端 SDK，全走 MCP（只读），不直连任何存储引擎。

---

## 1. 范围与核心决策

### 1.1 MVP 交付
1. **只读仪表板**（核心，完整可用）：组件健康、能力图、数据地图、知识库、Skills、记忆概况、经验记录、工作区上下文。
2. **Steward 对话窗**：UI + BFF 代理。Steward 未就绪时降级为"只读直连模式"（BFF 对自然语言做简单意图识别，调 MCP 只读资源回答），并明确标注模式；Steward 配置后代理至 Steward 获得完整管理能力。
3. **容器化**：单容器（BFF + 前端静态资源），docker-compose 编排，加入 `lakemind_lakemind` 网络。

### 1.2 关键决策
| 决策点 | 选择 | 理由 |
|--------|------|------|
| BFF 语言 | Python + FastAPI | 与 MCP 同生态，复用 `mcp` 客户端 SDK；方案§7.2 指定 fastapi |
| 前端 | Vue 3 + Vite + TypeScript + Pinia | 方案§7.3 指定 |
| 容器 | 多阶段：node 构建前端 → python:3.12-slim 跑 BFF 服务静态资源 | 单容器，产物自包含 |
| Monitor → MCP | streamable_http_client（只读资源 + 健康资源） | 全走 MCP，不直连存储 |
| 认证 | Monitor 持一个 **data-scope 只读 Token** | 只读纪律；无法调 admin 写工具 |
| 组件健康 | 新增 MCP 资源 `lake://system/health`（任何已认证用户可读） | 让 Monitor 用 data token 即可取健康，无需 admin scope，保持只读 |
| 租户视图 | MVP 单租户视图（Token 编码 tenant_id） | 多租户切换留演进 |

### 1.3 需 MCP 侧的小改动（一处）
在 LakeMindMCP 新增只读资源 `lake://system/health`，返回 `{s3, dragonfly, gravitino, embedding}` 状态，任何已认证用户可读（复用现有 `get_system_health` 逻辑，但不要求 admin scope）。这是让 Monitor 保持"只读 + data scope"的关键，符合方案"Monitor 依赖 MCP 健康检查接口"。

## 2. 技术选型

| 层 | 选型 |
|----|------|
| BFF | FastAPI + uvicorn + httpx + `mcp`（客户端 SDK） + prometheus-client |
| 前端 | Vue 3 + Vite + TypeScript + Pinia + Element Plus（组件库，快速出 MVP） |
| 图表 | ECharts（健康/用量可视化） |
| 容器 | 多阶段 Dockerfile（node:20-alpine 构建 → python:3.12-slim 运行） |

> 不引入任何数据引擎依赖。Monitor 镜像很小。

## 4. BFF API 设计

BFF 对前端暴露 REST，内部用 MCP 客户端读 MCP 资源。

| 方法 | 路径 | 说明 | 数据来源（MCP） |
|------|------|------|------------------|
| GET | `/api/health` | MCP 连通性 + BFF 自身 | MCP `/health` |
| GET | `/api/system-health` | 组件健康 | `lake://system/health` |
| GET | `/api/capabilities` | 能力图 | `lake://capabilities` |
| GET | `/api/workspace` | 工作区上下文 | `lake://workspace` |
| GET | `/api/data` | 数据集列表 | `lake://data` |
| GET | `/api/data/{name}` | 数据集详情 | `lake://data/{name}` |
| GET | `/api/knowledge` | 知识库列表 | `lake://knowledge` |
| GET | `/api/skills` | Skill 列表 | `lake://skills` |
| GET | `/api/memory` | 记忆概况 | `lake://memory` |
| GET | `/api/experience` | 经验记录 | `lake://experience` |
| POST | `/api/chat` | 对话 `{message}` → `{reply, mode}` | Steward 代理 / 只读直连降级 |
| GET | `/metrics` | Prometheus 指标 | BFF 自身 |

**MCP 客户端封装**：单例 `MCPReadClient`，持有只读 Token，复用一个 `ClientSession`（streamable HTTP），提供 `read_resource(uri)` 与 `call_tool(name, args)`。BFF 各端点调对应资源，聚合后返回前端友好结构。

**只读纪律**：BFF 仅调 Resource（read）+ `lake://system/health`，不调任何写工具。`/api/chat` 只读直连降级模式同样只读 MCP 资源。

## 5. 前端页面设计

- **Dashboard（总览）**：组件健康卡片（s3/dragonfly/gravitino/embedding 绿/红）+ 能力图 + 工作区 + 各资产计数。
- **Health**：组件健康详情 + MCP 连通性，自动 10s 刷新。
- **DataMap**：数据集表格（名称/列/行数），点击展开详情。
- **Knowledge / Skills / Memory / Experience**：对应资产列表/概况。
- **Chat**：对话窗，顶部标注当前模式（"Steward 已连接" / "只读直连模式（Steward 未就绪）"），消息列表 + 输入框。

布局：左侧导航 + 顶部状态栏 + 右侧内容区。Element Plus 组件库。

## 6. 与 MCP/Server 对接

```
浏览器 ──HTTP──> Monitor BFF(:3000) ──MCP(streamable_http)──> lakemind-mcp:8400
                                  └─(chat 代理)─> lakemind-steward:8910（可选，未就绪则降级）
```

- BFF 用 `mcp` 客户端 SDK 连 `http://lakemind-mcp:8400/mcp`，`Authorization: Bearer <read_token>`。
- 只读 Token：在 MCP `config.yaml` 增一条 `monitor` 身份（tenant=platform, scopes=[data]）。
- 健康走新增的 `lake://system/health` 资源（data scope 可读）。
- Steward 地址可配置；未配置/不可达时 `/api/chat` 走只读直连降级。

**只读直连降级**（Steward 未就绪时的 chat）：对用户消息做关键词意图识别——
- 含"健康/状态" → 返回 `lake://system/health`
- 含"数据/表" → 返回 `lake://data`
- 含"知识" → 返回 `lake://knowledge`
- 含"技能/skill" → 返回 `lake://skills`
- 含"记忆" → 返回 `lake://memory`
- 含"经验" → 返回 `lake://experience`
- 其他 → 提示该模式仅支持只读查询，并列举可识别意图

明确标注 `mode: "readonly-direct"`，不伪装成 Steward。

## 7. 容器化

### 7.1 Dockerfile（多阶段）
```dockerfile
# 阶段 1：前端构建
FROM node:20-alpine AS frontend
WORKDIR /fe
COPY frontend/ .
RUN npm ci && npm run build

# 阶段 2：BFF + 静态资源
FROM python:3.12-slim
WORKDIR /app
COPY pyproject.toml ./
COPY backend ./backend
COPY --from=frontend /fe/dist ./static
RUN pip install .
EXPOSE 3000
CMD ["python", "-m", "backend"]
```

### 7.2 docker-compose.yml
```yaml
services:
  lakemind-monitor:
    build: .
    container_name: lakemind-monitor
    environment:
      LAKE_CONFIG: /etc/lakemind/config.yaml
    ports: ["3000:3000"]
    volumes:
      - ./config/config.yaml:/etc/lakemind/config.yaml:ro
    networks: [lakemind]
    depends_on: [lakemind-mcp]   # MCP 由外部 compose 提供
    restart: unless-stopped
networks:
  lakemind:
    external: true
    name: lakemind_lakemind
```

> `depends_on` 跨 compose 项目不生效；MCP 由 LakeMindMCP 的 compose 启动。Monitor 启动时 BFF 健康检查会重试连 MCP。

### 7.3 config.example.yaml
```yaml
server:
  host: "0.0.0.0"
  port: 3000
mcp:
  url: "http://lakemind-mcp:8400/mcp"
  read_token: "<monitor-read-token>"     # data scope
steward:
  url: ""                                # 可选，留空则 chat 走只读直连降级
  token: ""
refresh:
  health_interval_sec: 10
```

## 8. 分阶段开发任务

| Phase | 内容 | 验收 |
|-------|------|------|
| **M0** | MCP 侧新增 `lake://system/health` 只读资源 + `monitor` 只读 Token；重建 MCP 容器 | MCP 资源可读、Token 可用 |
| **M1** | BFF 骨架：FastAPI + 配置 + MCP 客户端封装 + `/api/health` | `/api/health` 返回 MCP 连通 |
| **M2** | BFF 只读端点：capabilities/workspace/data/knowledge/skills/memory/experience/system-health | 各端点返回 MCP 资源数据 |
| **M3** | 前端骨架：Vue+Vite+Pinia+Element Plus，路由 + 布局 + API 层 | 页面可渲染 |
| **M4** | Dashboard + Health + DataMap + Knowledge + Skills + Memory + Experience 页面 | 各页面展示真实数据 |
| **M5** | Chat 对话窗：BFF `/api/chat`（Steward 代理 + 只读直连降级）+ 前端对话 UI | 降级模式可问答 |
| **M6** | 容器化：多阶段 Dockerfile + docker-compose + 静态资源服务 | 容器起、页面可访问 |
| **M7** | 端到端验证 `verify_monitor.py` + 文档 | **全量 PASS** |

### 里程碑
- **MA（M0–M2）**：BFF 经 MCP 拿到全部只读数据。
- **MB（M3–M4）**：仪表板完整展示真实运行态。
- **MC（M5–M7）**：对话窗 + 容器化 + 验证交付。

## 9. 验收标准

1. `docker compose up -d --build`（LakeMindServer + LakeMindMCP + LakeMindMonitor）全部健康。
2. 浏览器访问 `http://localhost:3000`：
   - Dashboard 显示 4 组件健康（s3/dragonfly/gravitino/embedding 全绿）、能力图 6 类、工作区、各资产计数。
   - DataMap 展示数据集列表与详情。
   - Knowledge/Skills/Memory/Experience 各自展示。
   - Chat：输入"健康"返回组件状态；顶部标注"只读直连模式"。
3. `python scripts/verify_monitor.py` 全 PASS，覆盖：
   - BFF 各 `/api/*` 端点返回有效数据
   - 组件健康 4 项 ok
   - 能力图含 6 类资产
   - 数据地图非空（对接已有数据）
   - chat 降级模式意图识别正确
   - MCP 不可达时 BFF 优雅降级（503 + 清晰提示）
4. Monitor 容器不包含任何数据引擎依赖（镜像小）。

## 10. 风险与演进

| 风险 | 应对 |
|------|------|
| node 镜像拉取失败 | 多阶段构建；若拉取失败，回退宿主机 `npm run build` 后拷 dist |
| MCP 不可达 | BFF 启动重试 + 前端展示降级态；`/api/health` 反映连通性 |
| Steward 未建 | chat 降级为只读直连模式，UI 明确标注；Steward 落地后填 url 即代理 |

演进预留：
- 多租户切换（Token 轮换 / 租户选择器）
- 审计日志与调用量（需 MCP 暴露审计 API）
- Prometheus 指标被外部 scrape（`/metrics` 已预留）
- Steward 就绪后 chat 自动切代理模式
