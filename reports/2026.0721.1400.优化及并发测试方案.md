## 2. S2: 向量检索优化

### 2.1 现状分析（已验证）

**L0 基准测试**（直接 Python 调 LanceDB，无 HTTP/API/锁）:

```
表: stress_vec_v2, 121,001 行, 768 维, 无索引
L0 单次搜索: 966~1760ms, 均值 1220ms
```

**结论**: 瓶颈在 LanceDB 引擎层，不是 API/锁/HTTP。121K 向量无 ANN 索引，暴力扫描。

### 2.2 执行步骤

#### Step 1: 创建 ANN 索引

```
tbl.create_index(
    num_partitions=256,      # 121K / 256 ≈ 473 vec/partition
    metric='L2',
    index_type='IVF_PQ',     # 默认类型, 压缩向量 + 快速近似搜索
    num_sub_vectors=16,      # 768 / 16 = 48 维/子向量
)
```

- 先在压测表上测试索引创建耗时、索引大小
- 验证 Recall@10（与暴力搜索结果对比）
- 如果 Recall < 0.95，调整参数（增大 num_partitions、改用 IVF_FLAT）

#### Step 2: L0/L1/L2 分层基准测试

| 层级 | 测试内容 | 隔离的变量 |
|------|----------|-----------|
| L0 | 直接 Python 调 LanceDB search | 引擎本身 |
| L1 | REST API `/vectors/{db}/{table}/search` | + HTTP + Pydantic + to_thread + Lock |
| L2 | Memory search（文本 → Embedding → LanceDB → 结果组装） | + Embedding + 业务逻辑 |

每层记录: `table_open_ms`, `search_ms`, `serialization_ms`, `queue_wait_ms`

#### Step 3: 缓存 Table Handle

当前 `lancedb.py` 每次操作都调 `self._conn(db).open_table(name)`，重复打开表有开销。

改为缓存 Table Handle:

```python
def _table(self, db: str, name: str):
    key = f"{db}/{name}"
    if key not in self._tables:
        self._tables[key] = self._conn(db).open_table(name)
    return self._tables[key]
```

注意: add 操作后可能需要刷新 handle（取决于 LanceDB 是否自动感知新数据）。

#### Step 4: 缩小锁粒度

当前: 一个 `threading.Lock` 锁住所有数据库的所有操作。

改为: **每表一把锁**，不同表的搜索可以并行。

```python
class LanceVectorStorage:
    def __init__(self, uri, **kwargs):
        self._uri = uri
        self._conns = {}
        self._tables = {}
        self._locks = {}          # {f"{db}/{name}": Lock}
        self._locks_guard = threading.Lock()  # 保护 _locks 字典本身

    def _get_lock(self, db, name):
        key = f"{db}/{name}"
        with self._locks_guard:
            if key not in self._locks:
                self._locks[key] = threading.Lock()
            return self._locks[key]
```

如果不同表仍触发 Rust panic，回退为全局锁但记录现象，后续用独立 Vector Executor 解决。

#### Step 5: 重跑 TC-5

目标:

```
单次搜索 p50 < 100ms
单次搜索 p99 < 500ms
5 并发 QPS > 10
10 并发错误率 < 1%
Recall@10 > 0.95
```

### 2.3 验收标准

- L0 搜索 p50 < 100ms（索引生效）

- Recall@10 > 0.95

## 3. S3: Memory 锁粒度修正与 Add 长尾定位

### 3.1 问题

当前 `BasicMemory` 的 8 个公开方法被整体加锁:

```python
for _name in ("add", "search", "get", "list_all", "update", "delete", "clear", "history"):
    _orig = getattr(BasicMemory, _name)
    def _wrapped(self, *args, **kwargs):
        with self._lock:
            return _orig(self, *args, **kwargs)
    setattr(BasicMemory, _name, _wrapped)
```

**后果**: 一个 11 秒的 LLM 调用（Memory Add 中的 `_extract_facts`）阻塞所有 Memory Search、List、Get。

### 3.2 执行步骤

#### Step 1: 删除全局方法锁

删除 `basic.py` 末尾的装饰器循环（第 444~451 行）。

#### Step 2: 精确锁定 LanceDB 临界区

只锁 LanceDB 操作，不锁 LLM/Embedding/PG/Redis:

| 方法 | 锁什么 | 不锁什么 |
|------|--------|----------|
| `add` | `conn.table_names()`, `conn.open_table()`, `tbl.add()`, `conn.create_table()` | `_extract_facts()` (LLM), `_embed()` (httpx), `_record_history()` (PG) |
| `search` | `conn.table_names()`, `conn.open_table()`, `tbl.search()` | `_embed()` (httpx), Redis scan |
| `get` | `conn.open_table()`, `tbl.search().where()` | — |
| `list_all` | `conn.open_table()`, `tbl.to_arrow()` | — |
| `update` | `conn.open_table()`, `tbl.delete()`, `tbl.add()` | `_embed()`, `_record_history()` |
| `delete` | `conn.open_table()`, `tbl.delete()` | `_record_history()` |
| `clear` | `conn.open_table()`, `tbl.to_arrow()`, `tbl.delete()` | Redis |
| `history` | 无 LanceDB 调用 | 全部不锁 |

实现方式: 在每个方法内部，用 `with self._lock:` 只包裹 LanceDB 调用行。LLM/Embedding/PG 调用在锁外执行。

#### Step 3: 缓存 LanceDB Connection

当前 `_connect_lance` 每次创建新连接:

```python
def _connect_lance(self, tenant_id, agent_id):
    import lancedb
    db_name = f"memory_{tenant_id}"
    conn = lancedb.connect(f"{self._lance_uri}/{db_name}")  # ← 每次新建
    return conn, f"mem_{agent_id}"
```

改为缓存:

```python
def _connect_lance(self, tenant_id, agent_id):
    db_name = f"memory_{tenant_id}"
    if db_name not in self._lance_conns:
        import lancedb
        self._lance_conns[db_name] = lancedb.connect(f"{self._lance_uri}/{db_name}")
    return self._lance_conns[db_name], f"mem_{agent_id}"
```

#### Step 4: Memory Add 长尾定位

拆分 TC-8 的 Add 测试为 5 个子测试:

| 子测试 | 内容 | 目标 |
|--------|------|------|
| `add.unique.infer_true` | 50 条完全不同的 Memory, infer=true | 记录每条 llm_extract_ms, embed_ms, dedup_ms, write_ms |
| `add.unique.infer_false` | 50 条不同 Memory, infer=false | 对比 infer=true 的耗时差 |
| `add.duplicate` | 50 条重复 Memory | 验证去重快速路径 |
| `add.cold` | 重启后首次 Add | 冷启动开销 |
| `add.warm` | 热态连续 Add | 稳态性能 |

在 `BasicMemory.add` 中增加分段计时（不改变返回值，计时结果写入日志或 debug 响应头）:

```python
t_llm = time.perf_counter()
facts = self._extract_facts(messages)
llm_ms = (time.perf_counter() - t_llm) * 1000

t_embed = time.perf_counter()
vec = self._embed(text)
embed_ms = (time.perf_counter() - t_embed) * 1000

# ... etc
```

#### Step 5: 重跑 TC-8

目标:

```
search p99 < 200ms
list p99 < 100ms
add (infer=false) p99 < 200ms
add (infer=true) p99 < 5s (LLM 调用本身)
10 并发 search QPS > 5
10 并发 search 错误率 < 5%
20 并发 search 错误率 < 10%
```

### 3.3 验收标准

- LLM 调用不再阻塞 Memory Search（并发验证）
- Add 长尾根因明确（LLM vs 去重 vs 冷启动）
- 10 并发 Memory Search 错误率 < 5%

```python
@router.post("/{db}/{name}/add_arrow")
async def add_vectors_arrow(db, name, request):
    # 1. Content-Type 校验
    if request.headers.get("content-type") != "application/x-arrow":
        raise HTTPException(415, "Unsupported media type")
    # 2. Content-Length 限制 (20K 向量 ≈ 60MB)
    cl = int(request.headers.get("content-length", 0))
    if cl > 100 * 1024 * 1024:
        raise HTTPException(413, "Payload too large, split into batches")
    body = await request.body()
    reader = pa.ipc.open_stream(body)
    data = reader.read_all()
    # 3. 维度校验
    vec_col = data.schema.field("vector").type
    if hasattr(vec_col, 'list_size') and vec_col.list_size != DIM:
        raise HTTPException(400, f"Expected dim={DIM}, got {vec_col.list_size}")
    # 4. 行数限制
    if data.num_rows > 20000:
        raise HTTPException(400, "Max 20K vectors per request")
    n = await asyncio.to_thread(_eng(request).add, db, name, data)
    return {"status": "ok", "rows_added": n}
```

## 5. 附加: 线程池治理

### 5.1 问题

当前所有同步调用共用 Python 默认线程池（`asyncio.to_thread` 默认 executor）。无背压、无资源隔离。

### 5.2 执行步骤

创建 `core/executor.py`:

```python
import asyncio
from concurrent.futures import ThreadPoolExecutor

class BlockingExecutor:
    def __init__(self):
        self._pools = {
            "s3": ThreadPoolExecutor(max_workers=16, thread_name_prefix="s3"),
            "lancedb": ThreadPoolExecutor(max_workers=2, thread_name_prefix="lancedb"),
            "pg": ThreadPoolExecutor(max_workers=8, thread_name_prefix="pg"),
            "valkey": ThreadPoolExecutor(max_workers=16, thread_name_prefix="valkey"),
            "iceberg": ThreadPoolExecutor(max_workers=4, thread_name_prefix="iceberg"),
        }
        self._semaphores = {
            "s3": asyncio.Semaphore(24),
            "lancedb": asyncio.Semaphore(2),
            "pg": asyncio.Semaphore(12),
            "valkey": asyncio.Semaphore(16),
            "iceberg": asyncio.Semaphore(4),
        }

    async def run(self, resource, fn, *args, **kwargs):
        async with self._semaphores[resource]:
            loop = asyncio.get_event_loop()
            return await loop.run_in_executor(
                self._pools[resource], lambda: fn(*args, **kwargs)
            )
```

在 `app.py` 初始化时挂到 `app.state.blocking_executor`。

API 层改为:

```python
# 旧: await asyncio.to_thread(_eng(request).put, bucket, key, body)
# 新: await request.app.state.blocking_executor.run("s3", _eng(request).put, bucket, key, body)
```

### 5.3 验收标准

- 不同资源的同步调用使用独立线程池
- 超过 Semaphore 限制时排队（不拒绝）
- 后续可扩展为返回 429/503

## 8. 风险与回退

| 风险 | 概率 | 回退方案 |
|------|------|----------|
| ANN 索引 Recall < 0.95 | 中 | 改用 IVF_FLAT（无压缩，精度更高）或增大 num_partitions |
| 每表锁仍触发 Rust panic | 中 | 回退为全局锁，记录现象，后续用独立 Vector Executor |
| 镜像重建 DNS 不通 | 高 | 移除 Dockerfile syntax 前缀，用内置 builder；或配置可用 mirror |
| Memory 锁细化后并发 panic | 低 | 回退为全局锁，但只锁 LanceDB 行（不锁 LLM/Embedding） |
| Arrow 端点 OOM | 低 | 限制单请求 20K 向量，超限返回 413 |
