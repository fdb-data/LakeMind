3. Ray 12 CPU 过度声明，挤占 ModelServing 资源

**判定**: 基线数据，脚本不可信，需修正。

### 2.3 v3: 第一轮优化 — 事件循环修复 (5 PASS / 4 FAIL)

**根因定位**: 逐层排查从压测数据到源码，发现 **async 端点直接调用同步阻塞 I/O** (boto3, LanceDB, Valkey)，冻结整个 asyncio 事件循环。一个请求做 S3 I/O 时，所有其他请求无法 accept。

**优化措施**:

#### P0: asyncio.to_thread — 同步 I/O 包线程化

9 个 API 文件的全部引擎调用包进 `await asyncio.to_thread(...)`:

| 文件 | 包了几个调用 | 引擎 |
|------|-------------|------|
| `api/objects.py` | 5 | SeaweedFS (boto3) |
| `api/vectors.py` | 6 | LanceDB |
| `api/kv.py` | 4 | Valkey |
| `api/memory.py` | 8 | BasicMemory |
| `api/tables.py` | 7 | Iceberg |
| `api/graph.py` | 5 | PostgreSQL graph |
| `api/search.py` | 1 | SearchService |
| `api/memories.py` | 8 | MemoryService v2 |
| `api/knowledge.py` | 5 | KnowledgeService |

####*无新依赖，无 API 契约变更。*

#### P1: uvicorn 多 worker

`__main__.py` 加 `workers` 参数。workers=2 在 Docker Desktop 下端口转发异常 (IPv6)，回退为 workers=1。P0 已足够。

#### P2: Arrow IPC 二进制向量入库端点

`api/vectors.py` 新增 `POST /{db}/{name}/add_arrow`，接受 Arrow IPC stream 二进制 body，跳过 Pydantic JSON 解析。

| 路径 | 10K vecs | 50K vecs | 加速比 |
|------|----------|----------|--------|
| JSON (原) | 715 vec/s | 670 vec/s | 1x |
| Arrow IPC (新) | 11988 vec/s | 14630 vec/s | ~20x |

#### 附加: LanceDB 全局锁

P0 后并发向量搜索触发 Rust panic (`pyo3_async_runtimes.RustPanic`)。加 `threading.Lock` 到 `lancedb.py` 全部方法和 `basic.py` 8 个公开方法。

**效果**:

| 指标 | v2 | v3 | 变化 |
|------|-----|-----|------|
| TC-1 文件上传 | 0% success | **100% success, 27 MB/s** | 根治 |
| TC-10 conc_10 | 53% error | **0% error** | 根治 |
| TC-10 conc_30 | 90% error | **0.67% error** | 根治 |
| TC-4 Arrow | — | **14630 vec/s** | 新端点 20x |

**遗留**: 全局锁串行化搜索 (QPS 0.7), Memory Add p99=11s (LLM 被锁阻塞)。

### 2.4 v4: 第二轮优化 — 索引 + 锁细化 (7 PASS / 2 FAIL)

**评审反馈**: v3 "有条件通过"，综合 6.8/10。事件循环已解决，但第二层瓶颈 (全局锁串行化) 刚暴露。要求: 索引 + 锁细化 + Memory Add 定位 + Arrow 全链路。

**优化措施**:

#### S1: 正式镜像重建

- 解决 Docker buildx DNS 问题
- 成功重建镜像 `docker buildx bake server-api --load`
- Git commit: `5981580`, Image: `sha256:b583210f44d5`

#### S2: 向量检索优化

**Step 1 — L0 基准验证**:

```
121,001 行, 768 维, 无索引
L0 直接 LanceDB: p50=1220ms (暴力扫描)
```

确认瓶颈在引擎层，不是 API/锁/HTTP。

**Step 2 — 创建 IVF_PQ 索引**:

```
tbl.create_index(num_partitions=128, metric='L2', index_type='IVF_PQ', num_sub_vectors=16)
耗时: 98.2s, 索引大小: 3.7MB
L0 with index: p50=14.2ms (86x 加速)
Recall@10: 1.0000
```

**Step 3 — Table Handle 缓存**:

`lancedb.py` 增加 `_tables` dict 缓存已打开的 Table Handle，避免重复 `open_table()` 调用。`add`/`create_table` 后 invalidate 缓存。

**Step 4 — 每表锁**:

`lancedb.py` 将全局锁改为 `_table_locks: dict[str, threading.Lock]`，每表独立锁。不同表的搜索可并行，同表搜索串行 (避免 Rust panic)。

**Step 5 — 索引刷新**:

发现批量 add 后新向量不被索引覆盖，搜索退化为暴力扫描。压测脚本在 TC-4 后自动 `create_index(replace=True)` 刷新索引。

**效果**:

| 指标 | v3 | v4 | 变化 |
|------|-----|-----|------|
| 向量搜索 L0 p50 | 1220ms | **14.2ms** | 86x |
| 向量搜索 L1 p50 | ~1400ms | **60ms** | 23x |
| 向量搜索 L1 p99 | — | **193ms** | — |
| 10 并发 QPS | 0.7 | **19.0** | 27x |
| 10 并发错误率 | 62% | **0%** | 根治 |

#### S3: Memory 锁粒度修正

**问题**: v3 的全局方法锁把 LLM/Embedding/PG/LanceDB 全部串行化。Memory Add 中 `_extract_facts` (LLM) 耗时 11s，阻塞所有 Search/List。

**修正**: 删除 `basic.py` 末尾全局装饰器，改为在每个方法内部精确锁定 LanceDB 临界区:

| 方法 | 锁 LanceDB | 不锁 |
|------|-----------|------|
| `add` | table_names, open_table, add, create_table | _extract_facts (LLM), _embed (httpx), _record_history (PG) |
| `search` | table_names, open_table, search | _embed (httpx), Redis scan |
| `get` | open_table, search().where() | — |
| `list_all` | open_table, to_arrow | — |
| `update` | open_table, delete, add | _embed, _record_history |
| `delete` | open_table, delete | _record_history |
| `clear` | open_table, to_arrow, delete | _record_history, Redis |
| `history` | 无 LanceDB | 全部不锁 |

同时缓存 LanceDB Connection (`_lance_conns` dict)。

**效果**:

| 指标 | v3 (全局锁) | v4 (精确锁) | 变化 |
|------|------------|------------|------|
| Add p99 | 11.1s | **33ms** | 335x |
| Search p50 | 95.6ms | 76.1ms | 1.3x |
| conc_10 QPS | 8.4 | **17.8** | 2.1x |
| conc_10 错误率 | 0% | 0% | — |

#### S4: Arrow 端点安全加固

`POST /{db}/{name}/add_arrow` 增加:
- Content-Type 校验: `application/x-arrow`
- Content-Length 限制: < 100MB
- 行数限制: ≤ 20,000 向量/请求
- 维度校验: 768 维

压测脚本将 50K 向量拆为 3 个 ~18K 批次。

## 4. 优化方法总结

### 4.1 方法论

```
压测暴露问题
→ 区分脚本 bug vs 系统问题
→ 修正脚本获取可信数据
→ 逐层排查: 压测数据 → 容器配置 → 源码
→ 定位根因
→ 分层优化: P0 根治 → P1 增强 → P2 突破
→ 每层改完立即验证
→ 评审反馈 → 下一轮
```

### 4.2 根因与修复对应关系

| 根因 | 现象 | 修复 | 轮次 |
|------|------|------|------|
| async 端点调同步 I/O → 事件循环冻结 | 10 并发即 53-100% 连接失败 | `asyncio.to_thread()` 包 49 个调用 | v3 |
| REST JSON 序列化 768 维向量 | 向量入库 700 vec/s | Arrow IPC 二进制端点 | v3 |
| LanceDB Rust 多线程 panic | 并发搜索 500 错误 | `threading.Lock` (先全局后每表) | v3→v4 |

| 全局锁串行化 LLM/Embedding/PG | Memory Add p99=11s | 精确锁只保护 LanceDB 临界区 | v4 |

## 6. 架构层面的启示

1. **async 端点 + 同步库 = 陷阱**: Python 生态中 boto3、LanceDB、psycopg2 都是同步库。在 FastAPI async 端点直接调用会阻塞事件循环。`asyncio.to_thread` 是通用解法。

2. **REST JSON 不是高维向量的最佳传输格式**: 768 维浮点 JSON 编解码占主要耗时。Arrow IPC 二进制格式零拷贝解析，20x 提升。

3. **ANN 索引是向量检索的必需品**: 121K 向量暴力扫描 1220ms，IVF_PQ 索引后 14ms。但索引需要在新数据写入后刷新。

4. **锁粒度决定并发能力**: 全局锁把不相关的操作 (LLM vs LanceDB) 串行化。精确锁只保护真正不线程安全的临界区，其余并发执行。

5. **压测要先验证脚本本身**: v1 的 6 个 FAIL 中有 2 个是脚本 bug。修正脚本后才能信任数据。

6. **分层基准测试定位瓶颈**: L0 (引擎) → L1 (API) → L2 (业务)。L0 1220ms 直接定位到索引缺失，而非 API 或锁。
