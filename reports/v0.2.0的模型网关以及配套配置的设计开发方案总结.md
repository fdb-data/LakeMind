# v0.2.0 模型网关及配套配置的设计开发方案总结

> 本文件是对 **当前已实现** 的模型网关（LakeMindModelServing）及其配套的 ControlCenter 模型管理功能的**现状梳理**，用于分析设计优化方向。
> 所有内容均基于 v0.2.0 实际代码与数据库状态，不是设计意图，是**已落地的实现**。

---

## 1. 一句话概括

当前模型网关由 **两套并行系统** 组成：
- **数据平面**：`LakeMindModelServing` (:10824) — 真正承载 LLM / Embedding / ASR 调用的网关，由 **静态 YAML** 在启动时配置 litellm Router。
- **控制平面**：`LakeMindServer` (:10823) 的 5 表模型注册表 + ControlCenter UI — 管理模型定义/部署/路由的元数据，但 **不在实际请求路径上**。

两者 **不同步**：UI 修改写入 Server 的 PG 表，不会传播到 ModelServing 的 litellm Router。

---

## 2. 三套子系统的职责与实际状态

### 2.1 子系统 A — LakeMindModelServing（真正的网关，:10824）

**代码位置**：`LakeMindModelServing/src/lakemind_model_serving/`

| 组件 | 文件 | 职责 |
|------|------|------|
| FastAPI app | `app.py` | 装配 gateway + registry + embedding_service + asr_service |
| litellm Router 封装 | `gateway.py` | 从 yaml providers 构建 litellm Router，支持运行时 register/deregister |
| 模型注册表 | `registry.py` | PG 表 `model_registry`（**单表**），持久化运行时注册的外部模型 |
| 本地 Embedding | `services/embedding.py` | fastembed（jina-embeddings-v2-base-zh, dim=768, ONNX CPU） |
| 本地 ASR | `services/asr.py` | faster-whisper（whisper-large-v3-turbo, INT8 CPU） |
| Chat API | `api/chat.py` | `POST /v1/chat/completions`（OpenAI 兼容） |
| Embedding API | `api/embeddings.py` | `POST /v1/embeddings`（本地优先，外部 fallback 到 gateway） |
| Audio API | `api/audio.py` | `POST /v1/audio/transcriptions`（本地 ASR） |
| Models 管理 API | `api/models.py` | `GET /v1/models`、`POST /v1/models/register`、`DELETE /v1/models/{id}` |
| 认证 | `auth.py` | 单一静态 Bearer Key（`MODELSERVING_API_KEY`） |

**配置来源**：`config/models.yaml`（静态文件，启动时读取，不热加载）

```yaml
gateway:
  default_chat_model: "deepseek-v4-flash"
  default_embed_model: "jina-embeddings-v2-base-zh"
  fallback:
    chat: ["deepseek-v4-flash"]
    embed: ["jina-embeddings-v2-base-zh"]

llm_providers:
  - name: "modelarts"
    type: "openai"
    base_url: "https://api.modelarts-maas.com/openai/v1"
    api_key: "${MAAS_API_KEY}"
    models:
      - id: "deepseek-v4-flash"
        litellm_model: "openai/deepseek-v4-flash"
        context: 64000
        tags: [chat]

embedding:
  built_in:
    provider: "fastembed"
    model: "jinaai/jina-embeddings-v2-base-zh"
    dim: 768

asr:
  built_in:
    provider: "faster-whisper"
    model_alias: "whisper-small"
    model_path: "/models/asr/faster-whisper-small"
```

**实际运行状态**：
- litellm Router 启动时加载 1 个 provider（modelarts）+ 1 个 chat 模型
- `model_registry` 表 **0 行**（运行时注册功能未被使用）
- 所有路由决策由 yaml 的 `default_chat_model` / `default_embed_model` 决定

### 2.2 子系统 B — LakeMindServer 模型管理（控制平面，:10823）

**代码位置**：`LakeMindServer/src/lakemind_server/`

| 组件 | 文件 | 职责 |
|------|------|------|
| Service | `services/model_management_service.py` | 5 表 CRUD + enable/disable + test_deployment + resolve_profile + import_from_yaml |
| API | `api/models.py` | 14 个端点（definitions/deployments/profiles/routes 的 CRUD + enable/disable/test/resolve） |
| Migration | `migrations/versions/006_model_management.py` | 创建 5 张表 |
| Bootstrap | `scripts/bootstrap.py` | 启动时从 models.yaml 导入到 5 表 |

**数据库 schema（5 表，过度规范化）**：

```
model_definitions (5 rows)
  ├── model_deployments (8 rows, 1:N)
  │     └── model_routes (7 rows, N:1 from profile side)
  ├── embedding_spaces (1 row)
  └── model_profiles (5 rows)
        └── model_routes (profile_name → deployment_id 映射)
```

| 表 | 行数 | 列 | 用途 |
|----|------|-----|------|
| `model_definitions` | 5 | model_id, name, model_type, capabilities, provider_family, context_length, embedding_dim, modalities, metadata | 模型逻辑定义 |
| `model_deployments` | 8 | deployment_id, model_id, provider, **endpoint**, secret_ref, status, priority, timeout_ms, max_concurrency, health_status, desired_state, test_state | 物理部署实例 |
| `model_profiles` | 5 | profile_id, name, description | 命名路由组 |
| `model_routes` | 7 | route_id, profile_name, deployment_id, priority, is_fallback, tenant_id | profile→deployment 映射 |
| `embedding_spaces` | 1 | space_id, model_id, model_revision, dimension, normalize, distance_metric | 嵌入空间注册 |

**实际数据**：

```
definitions: deepseek-v4-flash(chat), jinaai/jina-embeddings-v2-base-zh(embedding),
             iic/SenseVoiceSmall(asr), whisper-large-v3-turbo(asr),
             jina-embeddings-v2-base-zh(embedding)

deployments: 8 个（5 enabled + 3 disabled）
  - 5 enabled: 全部指向 lakemind-model-serving:10824 的不同路径
  - 3 disabled: 指向 localhost:10824（容器内不可达，已手动禁用）

profiles: default, meeting-asr, meeting-embedding, meeting-knowledge-extract, meeting-minutes

routes: 7 条，将 profile 映射到 deployment
```

**关键函数**：

- `resolve_profile(profile_name, tenant_id)` — 联合查 routes→deployments→definitions，返回 `{endpoint, model_id, fallbacks}`。**但实际消费者不调用此函数**。
- `test_deployment(deployment_id)` — Server 侧发 HTTP 探测请求到 deployment.endpoint，更新 health_status。**控制平面操作**。
- `enable/disable_deployment()` — 更新 status + desired_state，写 outbox 事件。**但不会通知 ModelServing**。

### 2.3 子系统 C — ControlCenter ModelServing 页面（UI）

**代码位置**：`LakeMindControlCenter/frontend/src/pages/ModelServing.tsx`（410 行）

**3 个 Tab**：

| Tab | 内容 | 操作 |
|-----|------|------|
| Model Definitions | 5 行表 | Add / Edit |
| Deployments | 8 行表 | Add / Edit / Enable / Disable / **Test** |
| Profiles & Routes | 2 张表 | Add Profile / Add Route / Delete Route |

**4 个 Modal**：Add/Edit Definition、Add/Edit Deployment、Add Profile、Add Route

**BFF 代理**：`bff/app.py` 的 `/models/{path:path}` → Server `/api/v1/models/{path}`

**Test 功能**：点击 Test 按钮 → Server `test_deployment` → HTTP 探测 endpoint → 返回 `{success, latency_ms, status_code, response_preview}` → 更新 health_status

## 3. 实际请求路径（运行时数据流）

这是 **真正发生调用时的路径**，与上述控制平面无关：

```
Meeting Agent Skill (Ray job)
  │  lakemind_utils.py 硬编码:
  │    MS_URL = env["MODEL_SERVING_URL"]  # http://lakemind-model-serving:10824
  │    model = "deepseek-v4-flash"        # 硬编码模型名
  │
  ├─ llm_chat() ──→ POST MS_URL/v1/chat/completions  {model: "deepseek-v4-flash"}
  ├─ asr()       ──→ POST MS_URL/v1/audio/transcriptions
  └─ embedding   ──→ POST MS_URL/v1/embeddings       {model: "jina-embeddings-v2-base-zh"}
         │
         ▼
  LakeMindModelServing (:10824)
    ├─ chat.py    → gateway.chat(model="deepseek-v4-flash")
    │                → litellm Router.completion(model="openai/deepseek-v4-flash")
    │                → ModelArts API
    ├─ embeddings.py → 本地 fastembed（硬编码 jina 模型）
    └─ audio.py   → 本地 faster-whisper
```

**关键观察**：
1. **消费者直连 ModelServing**，不经过 Server 的 resolve_profile
2. **模型名硬编码在 Skill 代码中**（`model = "deepseek-v4-flash"`），不通过 profile 查询
3. **Profile/Route 表未被运行时消费**，仅用于 UI 展示和 test_deployment
4. **Server 的 5 表是纯控制平面元数据**，不影响实际请求路由

## 4. 配置入口汇总

模型配置实际有 **3 个入口**，互不同步：

| 入口 | 作用域 | 影响什么 | 何时生效 |
|------|--------|----------|----------|
| `LakeMindModelServing/config/models.yaml` | ModelServing 容器 | litellm Router 的 provider/model/fallback | **重启容器时** |
| `LakeMindServer` PG 5 表（通过 UI 或 bootstrap） | Server 容器 | UI 展示、test_deployment、resolve_profile | **立即**（但不影响路由） |
| `examples/meeting-agent/.env` 的 `MODEL_SERVING_URL` | Meeting Agent 容器 | Skill 代码调用的目标地址 | **重启容器时** |

**后果**：
- 在 UI 里新增一个 Deployment，ModelServing 的 litellm Router **不会** 加载它
- 在 UI 里禁用一个 Deployment，ModelServing 的 litellm Router **继续** 路由到它
- 修改 models.yaml 后需重启 ModelServing 容器，UI 不会自动反映变更

## 5. 当前实现的问题清单

### 5.1 架构层面

| # | 问题 | 严重度 |
|---|------|--------|
| 1 | **两套并行注册表不同步** — ModelServing 的 `model_registry` 表（0 行）与 Server 的 5 表（21 行）是两套独立系统 | 高 |
| 2 | **控制平面不影响数据平面** — UI 操作写入 Server DB，不传播到 ModelServing Router | 高 |
| 3 | **Profile/Route 是空转层** — resolve_profile 存在但无消费者调用；实际路由由硬编码模型名决定 | 高 |
| 4 | **5 表过度规范化** — 对 5 个模型而言，definitions + deployments + profiles + routes + embedding_spaces 是过度设计 | 中 |
| 5 | **endpoint 字段冗余** — 所有 enabled deployment 都指向 `lakemind-model-serving:10824` 的不同路径，endpoint 本可由 model_type 推导 | 中 |
| 6 | **embedding_spaces 表未消费** — 1 行数据，无代码在运行时查询它 | 低 |

### 5.2 运维层面

| # | 问题 | 根因 |
|---|------|------|
| 7 | **localhost vs 容器名** — 出现 3 个 `localhost:10824` 的 disabled deployment | endpoint 手动填写，无校验 |
| 8 | **重复 deployment** — 同一模型有 2 个 deployment（localhost + 容器名） | bootstrap 创建 + 手动/UI 创建无去重 |
| 9 | **模型名不一致** — `jinaai/jina-embeddings-v2-base-zh`（yaml 原名）vs `jina-embeddings-v2-base-zh`（别名）两个 definition 并存 | bootstrap 从 yaml 导入 + UI 手动创建 |
| 10 | **配置变更需重启** — 修改 models.yaml 后需重启 ModelServing 容器 | 无热加载机制 |

### 5.3 用户体验层面

| # | 问题 |
|---|------|
| 11 | **新增模型需 3-4 步** — Add Definition → Add Deployment → (Add Profile → Add Route)，无引导式流程 |
| 12 | **Definition 与 Deployment 关系不直观** — UI 上分两个 Tab，需切换才能看到关联 |
| 13 | **Test 结果是一次性快照** — 无定期健康检查，health_status 只在点 Test 时更新 |
| 14 | **无模型用量/成本监控** — 不记录哪个模型被调了多少次、latency、token 消耗 |

## 9. 与上层消费者的关系

### 9.1 Meeting Agent（当前唯一真实消费者）

```python
# lakemind_utils.py（Skill 代码，运行在 Ray job 中）
MS_URL = os.environ.get("MODEL_SERVING_URL", "http://lakemind-model-serving:10824")
MS_KEY = os.environ.get("MODELSERVING_API_KEY", "lakemind-modelserving-key")

def llm_chat(system_prompt, user_content, model="deepseek-v4-flash"):
    httpx.post(f"{MS_URL}/v1/chat/completions", ...)

def asr(audio, filename):
    httpx.post(f"{MS_URL}/v1/audio/transcriptions", ...)

def embedding(texts):
    httpx.post(f"{MS_URL}/v1/embeddings", ...)
```

**特点**：
- 直连 ModelServing，不经过 Server
- 模型名硬编码（`deepseek-v4-flash`）
- 不查询 profile/route
- 不使用 Server 的模型管理 API

### 9.2 Server 内部消费者

Server 的 `cognitive.memory` 引擎需要 embedding 和 LLM（用于记忆事实抽取），通过 `MODEL_SERVING_URL` 环境变量调用 ModelServing，同样 **不经过 5 表**。

### 9.3 Steward（ControlCenter 内嵌）

Steward 通过 ModelServing LLM 驱动对话，同样直连 :10824。

## 10. 设计决策追溯

以下是 v0.2.0 开发过程中做出的关键设计决策（及其代价）：

| 决策 | 动机 | 代价 |
|------|------|------|
| 引入 5 表模型管理 | 想做 Definition/Deployment 分离 + Profile 路由 + 多租户 | 5 表 + 439 行 service + 139 行 API，但运行时不消费 |
| ModelServing 用静态 yaml | litellm Router 需要启动时初始化 | 配置变更需重启 |
| Server 做 test_deployment | 想在 UI 上点 Test 按钮检测连通性 | Server 侧发 HTTP 探测，跨平面操作 |
| endpoint 存全 URL | 想支持外部 provider | 所有内部 deployment 都存重复的长 URL |
| Profile/Route 支持 tenant_id | 想做多租户模型路由 | 当前无租户级路由需求 |
| embedding_spaces 表 | 想做 reindex 安全保障 | 1 行数据，无消费方 |

## 11. 可优化方向（供分析参考，不做方案建议）

以下仅列出 **可考虑的优化方向**，具体方案需进一步分析：

1. **统一注册表** — 消除 ModelServing 的 `model_registry` 与 Server 的 5 表之间的二元性
2. **让控制平面影响数据平面** — UI 变更 → 通知 ModelServing 热加载（已有 register_model 能力，但未接通）
3. **简化表结构** — 评估 5 表是否可合并为 2-3 表（如 definitions + deployments 合为 models，profiles + routes 合为 routing_rules）
4. **消除空转层** — 要么让消费者真正调用 resolve_profile，要么去掉 profile/route
5. **endpoint 由 model_type 推导** — 内部模型不需要存全 URL，只存 provider + model_type 即可拼出 endpoint
6. **配置热加载** — ModelServing 监听 PG 变更或接收 webhook，无需重启
7. **UI 引导式流程** — 新增模型时一步到位（填 provider + key → 自动创建 definition + deployment）
8. **定期健康检查** — 替代手动 Test 按钮，后台定期探测

## 13. 总结

当前模型网关的核心矛盾是 **控制平面与数据平面脱节**：

- **数据平面**（ModelServing + litellm）设计简洁、运行可靠，但由静态 yaml 驱动
- **控制平面**（Server 5 表 + UI）功能完整、表结构规范，但不影响实际路由
- 两者各自正确，但 **没有形成闭环**

实际效果：UI 上的模型管理是 **展示性的**，真正改路由需要改 yaml + 重启容器。这导致 5 表 + 439 行 service + 410 行 UI 的投入未产生对应的运行时价值。
