## 2. 问题分析

### 2.1 分析方法

逐层排查，从压测数据到容器配置到源码：

1. **看错误类型**：所有并发失败的状态码都是 `0`（连接被拒绝/超时），不是 4xx/5xx。这意味着不是应用层返回的错误，而是连接层根本建不起来。
2. **看容器配置**：`docker inspect` 看 server-api 的启动命令，发现 `python -m lakemind_server`，追到 `__main__.py`。
3. **读源码**：`__main__.py` 第 9 行 `uvicorn.run(...)` 没有传 `workers` 参数——单 worker 单进程。但这不是根因，单 worker 也能处理并发。
4. **读 API 层代码**：这是关键一步。逐个读 `api/objects.py`、`api/vectors.py`、`api/memory.py` 等，发现所有 async 端点里直接调用同步阻塞函数：

```python
# objects.py — 优化前
@router.put("/{bucket}/{key:path}")
async def put_object(bucket, key, request):
    body = await request.body()
    _eng(request).put(bucket, key, body)  # ← boto3 同步阻塞！
    return {"status": "ok"}
```

```python
# vectors.py — 优化前
@router.post("/{db}/{name}/search")
async def search_vectors(db, name, body, request):
    results = _eng(request).search(...)  # ← LanceDB 同步阻塞！
    return {"results": results}
```

### 2.2 根因定位

**根因：async 端点中直接调用同步阻塞 I/O，冻结整个 asyncio 事件循环。**

FastAPI 的 async 端点运行在 uvicorn 的 asyncio 事件循环上。当一个请求调用 `boto3.put_object()` 或 `LanceDB.search()` 时，这些同步函数会阻塞当前线程。由于 asyncio 是单线程事件循环，一个请求阻塞时，所有其他请求（包括新连接的 accept）都无法处理。

这解释了所有现象：
- TC-1 20 并发上传：第一个请求在做 S3 I/O 时，其余 19 个连接全部排队超时
- TC-5 10 并发检索：第一个 LanceDB search 阻塞事件循环，其余 9 个超时
- TC-10 阶梯并发：混合操作中任何一个阻塞，全部雪崩
- TC-2 单文件读写、TC-12 单条调用：串行不触发并发，所以正常

### 2.3 向量入库瓶颈分析

向量入库 ~700 vec/s 的瓶颈是 REST API 的 JSON 序列化。768 维浮点向量通过 Pydantic JSON 解析 → `pa.Table.from_pylist()` 转换，CPU 大量时间花在 JSON 编解码上。LanceDB 引擎本身（L0 层）的写入能力远高于此。

### 2.4 次生问题：LanceDB 线程安全

P0 修复后并发搜索不再超时，但出现新错误：`pyo3_async_runtimes.RustPanic: rust future panicked: unknown error`。这是 LanceDB 的 Rust 核心（PyO3 绑定）在多线程并发访问时 panic。`asyncio.to_thread` 把同步调用丢到线程池，多个线程同时操作 LanceDB 触发了 Rust 侧的线程安全 bug。

## 3. 优化措施

按 P0→P1→P2 三层执行，每层改完立即验证。

### 3.1 P0: 同步 I/O 包 `asyncio.to_thread()`

**原理**: `asyncio.to_thread(fn, *args)` 将同步函数 `fn` 丢到默认线程池执行，立即返回一个 awaitable。事件循环不被阻塞，可以同时 accept 和处理其他请求。

**改动**: 9 个 API 文件，全部 `_eng(request).xxx()` 调用改为 `await asyncio.to_thread(_eng(request).xxx, ...)`：

| 文件 | 包了几个调用 | 覆盖的引擎 |
|------|-------------|-----------|
| `api/objects.py` | 5 | SeaweedFS (boto3) |
| `api/vectors.py` | 6 | LanceDB |
| `api/kv.py` | 4 | Valkey (redis) |
| `api/memory.py` | 8 | BasicMemory (LanceDB + PG + httpx) |
| `api/tables.py` | 7 | Iceberg |
| `api/graph.py` | 5 | PostgreSQL graph |
| `api/search.py` | 1 | SearchService |
| `api/memories.py` | 8 | MemoryService (v2) |
| `api/knowledge.py` | 5 | KnowledgeService |

无新依赖，无 API 契约变更，纯内部实现优化。

### 3.2 P1: uvicorn 多 worker

**改动**: `__main__.py` 加 `workers` 参数从环境变量读取，`docker-compose.yml` 加 `UVICORN_WORKERS` 环境变量。

**结果**: workers=2 时 Docker Desktop 的端口转发出现 IPv6 绑定问题（`localhost` 解析到 `::1`，Docker proxy 的 IPv6 转发异常），host 侧无法连接。回退为 workers=1。P0 的 `to_thread` 已足够解决并发问题，多 worker 是锦上添花，暂不强求。

### 3.3 P2: Arrow IPC 二进制向量入库端点

**改动**: `api/vectors.py` 新增 `POST /{db}/{name}/add_arrow` 端点：

```python
@router.post("/{db}/{name}/add_arrow")
async def add_vectors_arrow(db, name, request):
    body = await request.body()
    reader = pa.ipc.open_stream(body)
    data = reader.read_all()
    n = await asyncio.to_thread(_eng(request).add, db, name, data)
    return {"status": "ok", "rows_added": n}
```

客户端直接发送 Arrow IPC stream 二进制 body，跳过 Pydantic JSON 解析和 `pa.Table.from_pylist()` 转换。

**效果**: 10K 向量入库从 13.9s → 0.83s，吞吐从 715 vec/s → **11988 vec/s**（17x）；50K 从 74.6s → 3.4s，**14630 vec/s**（21x）。

### 3.4 附加: LanceDB 线程锁

**问题**: P0 后多线程并发访问 LanceDB 触发 Rust panic。

**修复**:
- `lancedb.py`: `__init__` 加 `self._lock = threading.Lock()`，所有公开方法加 `with self._lock:`
- `basic.py`: `__init__` 加 `self._lock = threading.Lock()`，文件末尾用循环装饰 8 个公开方法

锁将 LanceDB 操作序列化，消除 Rust panic。代价是向量搜索 QPS 降为 1/search_time（~0.7 QPS），但消除了 500 错误。

## 5. 初步分析

### 5.1 已解决的核心问题

**并发连接瓶颈已根治。** 优化前 10 并发即大面积失败，优化后 30 并发 0.67% 错误率。根因是 async 端点里直接调同步阻塞 I/O，用 `asyncio.to_thread` 解决。这是本次优化最重要的成果——系统从"单请求可用、并发不可用"变为"并发可用"。

**向量入库吞吐瓶颈已突破。** JSON 路径 ~700 vec/s 是 REST API 固有限制（768 维浮点 JSON 编解码），Arrow IPC 二进制端点达到 ~15000 vec/s，20 倍提升。客户端改用 Arrow 路径即可。

### 5.2 尚存的限制

**向量检索单次延迟高（~1.4s）。** 60K+ 768 维向量无 IVF 索引，LanceDB 做暴力扫描。加上锁序列化，高并发时后续请求排队超时。解决方向：
- 创建 LanceDB IVF 索引（`tbl.create_index(num_partitions=...)`），单次搜索可降至 10~50ms
- 或接受当前性能，定位为"低并发高精度检索"场景

**LanceDB 多线程 Rust panic。** 用锁绕过，但牺牲了并发搜索能力。这是 LanceDB PyO3 绑定的已知问题，未来可关注版本更新或改用 async API。

**uvicorn 多 worker 在 Docker Desktop 下端口异常。** IPv6 绑定问题，非应用代码问题。生产环境（Linux）应无此问题，可正常使用 workers=2+。

### 5.3 架构层面的启示

1. **async 端点 + 同步库 = 陷阱。** Python 生态中 boto3、LanceDB、psycopg2 等都是同步库。在 FastAPI async 端点中直接调用会阻塞事件循环。`asyncio.to_thread` 是通用解法，但每个端点都要改。可考虑用 `run_in_executor` 装饰器统一处理。

2. **REST JSON 不是向量数据的最佳传输格式。** 768 维浮点向量 JSON 编码体积大、解析慢。Arrow IPC 二进制格式体积小 3~5 倍、零拷贝解析，适合批量向量传输。

3. **压测要先验证脚本本身。** 首轮压测 6 个 FAIL 中有 2 个是脚本 bug（speedup 计算、冷启动未重启）。修正脚本后才能信任数据。

### 5.4 后续建议

| 优先级 | 事项 | 预期收益 |
|--------|------|----------|
| P0 | 客户端向量入库改用 Arrow IPC 端点 | 入库吞吐 700→15000 vec/s |
| P1 | LanceDB 创建 IVF 索引 | 检索延迟 1.4s→~30ms, 并发 QPS 大幅提升 |
| P1 | TC-1 阈值调至 25 MB/s 或增大并发至 50 | 压测通过 |
| P2 | 记忆 add 操作支持 infer=false 跳过 LLM | add p99 11s→~20ms |
| P2 | 生产环境 (Linux) 启用 uvicorn workers=2~4 | 吞吐上限提升 |
